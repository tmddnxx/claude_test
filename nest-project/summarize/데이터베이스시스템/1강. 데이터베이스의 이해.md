# 데이터베이스의 이해 (1강)

## 1. 전체 개요
이 강의는 데이터베이스 시스템 과목의 첫 번째 강의로, 데이터 관리의 필요성과 데이터베이스의 역할을 소개한다. 전통적인 파일 처리 시스템의 4가지 한계(데이터 종속, 중복, 무결성 훼손, 동시 접근 문제)를 설명하고, 데이터베이스가 이를 어떻게 해결하는지 다룬다. 마지막으로 데이터베이스 언어(DDL/DML)와 시스템 아키텍처(중앙 집중 방식, 클라이언트-서버 방식)를 개괄한다.

## 2. 핵심 개념

- **데이터 생성 트렌드**: 레거시 시스템, 스마트 디바이스, 소셜 미디어, IoT의 4가지 변화로 제타바이트 규모의 데이터가 생성됨
- **파일 처리 시스템**: 각 애플리케이션이 고유한 파일을 직접 저장·관리하는 전통적 방식
- **데이터베이스의 패러다임 전환**: 데이터 사용 영역과 데이터 관리 영역을 분리하여 DBMS가 브로커 역할 수행
- **DBMS 3단계 구조**: 외부 단계(뷰/외부 스키마), 개념 단계(개념 스키마), 내부 단계(내부 스키마)
- **트랜잭션**: 하나의 논리적 작업을 처리하기 위한 일련의 데이터베이스 명령 집합
- **메타데이터**: 값에 의미를 부여하는 데이터에 대한 데이터 (시스템 카탈로그/데이터 사전에 저장)

## 3. 중요 정의

| 용어 | 정의 |
|------|------|
| **데이터** | 어떤 사실에 대한 정량적·정성적 특징을 나타낼 수 있는 값과 값의 의미를 설명하는 메타데이터의 결합 |
| **데이터베이스** | 특정 기관의 애플리케이션 시스템에서 사용되는 데이터의 집합 (소프트웨어가 아닌 주제지향적 데이터 집합) |
| **DBMS** | 데이터베이스에 저장된 데이터의 구성, 저장, 관리, 사용을 위한 소프트웨어 패키지 |
| **데이터베이스 시스템** | DBMS를 포함하여 사용자에게 정보를 제공하는 컴퓨터 기반의 총체적 시스템 |
| **트랜잭션** | 하나의 논리적 작업을 처리하기 위한 일련의 데이터베이스 명령의 집합 |
| **DDL (데이터 정의 언어)** | 데이터베이스 객체(테이블, 인덱스, 뷰 등)를 생성·수정·삭제하기 위한 언어 |
| **DML (데이터 조작 언어)** | 구조화된 데이터를 검색·삽입·삭제·수정할 수 있도록 지원하는 언어 |
| **무결성** | 데이터의 값이 정해진 제약 조건을 만족하는 정확한 상태 |

**파일 처리 시스템의 4가지 문제:**
1. **데이터 종속** — 물리적 종속(하드웨어 변경 시 동작 불가) + 논리적 종속(데이터 형식 변경 시 동작 불가)
2. **데이터 중복** — 일관성, 보안성, 경제성 문제 유발
3. **무결성 훼손** — 제약 조건 검증 불가 (예: 최대 18학점인데 24학점 저장)
4. **동시 접근** — 다수 사용자의 동시 접근 시 일관성 훼손

**데이터베이스 4가지 특징:**
1. **자기기술성**(Self-describing) — 메타데이터를 함께 관리
2. **프로그램-데이터 독립/추상화** — 물리적 위치를 몰라도 개념적 요청만으로 검색 가능
3. **다중 뷰 제공** — 사용자별 필요한 데이터만 접근 (보안성 확보)
4. **다수 사용자 트랜잭션 처리** — 동시 접근 시 일관성 유지

## 4. 시험 예상 포인트

- **데이터 vs 값 vs 메타데이터 구분**: 값(12) 자체는 데이터가 아니며, 메타데이터(오늘, 낮 최고 기온)가 결합되어야 데이터
- **데이터·데이터베이스·DBMS·데이터베이스 시스템 4개 용어의 정확한 구분** (정의 비교 문제 출제 가능성 높음)
- **파일 처리 시스템의 4가지 문제점**과 각각의 구체적 사례
- **물리적 데이터 종속 vs 논리적 데이터 종속** 구분
- **데이터 중복이 일관성·보안성·경제성에 미치는 영향**
- **DBMS 3단계 구조**(외부-개념-내부)와 각 단계의 스키마 역할
- 개념↔내부 사상 → 물리적 데이터 독립성 / 개념↔외부 사상 → 논리적 데이터 독립성
- **트랜잭션 개념과 동시 접근 문제 해결 원리** (은행 계좌 예시)
- **DDL vs DML**의 역할 차이
- **중앙 집중 방식 vs 클라이언트-서버 방식(2계층/3계층)**의 장단점 비교

## 5. 한 페이지 요약

현대 사회에서는 스마트 디바이스, 소셜 미디어, IoT 등의 확산으로 제타바이트 규모의 데이터가 생성되고 있으며, 이를 효율적으로 저장·검색·관리하는 것이 핵심 과제다.

전통적 **파일 처리 시스템**은 각 애플리케이션이 고유 파일을 직접 관리하는 방식으로, **데이터 종속**(하드웨어·형식 변경 시 동작 불가), **데이터 중복**(일관성·보안성·경제성 저해), **무결성 훼손**(제약 조건 검증 불가), **동시 접근 문제**(다수 사용자 접근 시 일관성 파괴)라는 4가지 근본적 한계를 가진다.

**데이터베이스**는 이를 해결하기 위해 데이터 사용 영역과 관리 영역을 분리한다. DBMS가 애플리케이션과 저장소 사이에서 브로커 역할을 수행하며, 4가지 핵심 특징을 제공한다: (1) **자기기술성** — 메타데이터(시스템 카탈로그)를 통해 데이터의 의미를 관리, (2) **프로그램-데이터 추상화** — 3단계 구조(외부·개념·내부 스키마)를 통해 물리적·논리적 데이터 독립성 확보, (3) **다중 뷰** — 사용자별 권한에 맞는 데이터만 제공하여 보안성 확보, (4) **트랜잭션 처리** — 논리적 작업 단위를 묶어 동시 접근 시에도 일관성 유지.

용어를 정확히 구분해야 한다: **데이터**(값+메타데이터), **데이터베이스**(주제지향적 데이터 집합), **DBMS**(데이터 관리 소프트웨어 패키지), **데이터베이스 시스템**(DBMS+애플리케이션을 포함한 총체적 시스템).

데이터베이스 언어는 객체를 정의하는 **DDL**과 데이터를 조작(검색·삽입·수정·삭제)하는 **DML**로 구분되며, 현재 **SQL**이 세계 표준이다. 시스템 아키텍처는 **중앙 집중 방식**(단일 서버+터미널, 구축 간편하나 과부하 위험)과 **클라이언트-서버 방식**(2계층/3계층, 부하 분산 가능하나 구축 비용 높음)으로 나뉘며, 상황에 따라 적절한 방식을 선택해야 한다.
